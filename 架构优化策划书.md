# 架构优化策划书

## 1. 总体判断

当前系统不需要推翻重写。更稳妥的路线是保留现有 `FastAPI` 单体后端和原生 H5 页面，在此基础上逐步把缓存、任务状态、核心业务数据和长任务执行拆出去。

你的优化方向是正确的，尤其是下面这条顺序：

1. 先用小改动提升稳定性：缓存、限流、轮询降频、统一日志和错误处理。
2. 再引入 Redis：承接热点缓存、job 状态、短期任务结果。
3. 再引入 PostgreSQL：保存核心业务数据。
4. 最后把 AI、筛选、回测等重任务迁到 Worker。
5. 前端先拆文件和减少 DOM 压力，不急着切 React/Vue。

需要明确边界：

- Redis 适合保存热数据、短期缓存、任务状态、临时结果，不保存长期核心业务数据。
- PostgreSQL 适合保存用户配置、持仓、订单、AI 推荐历史、风控记录、任务历史、策略记忆和模型配置。
- 行情、K线、新闻、板块等第三方数据应由后端定时拉取并缓存，前端只读后端结果，不能让每个用户请求都直接触发第三方 API。
- Worker 应在 Redis job 状态稳定后再引入，避免一步改太大。

## 2. 当前架构现状

当前主网页是 `FastAPI + 原生 H5 单页`：

- 后端主入口是 `backend/app/main_simple.py`。
- 前端主页面是 `backend/static/index.html`。
- 业务逻辑集中在 `backend/app/services/`。
- 页面通过 `/api/v1/...` 接口和后端交互。

当前主要问题：

- `main_simple.py` 承担了太多接口、任务状态和启动逻辑，文件越来越难维护。
- `index.html` 同时承载 UI、状态、API、图表、任务轮询和业务逻辑，拆分压力已经明显。
- 一部分运行状态仍保存在内存变量中，例如 job 状态、筛选任务、AI 任务、实时状态。
- 核心业务数据有一部分保存在内存或本地 JSON 文件中，服务重启后存在恢复、同步和一致性问题。
- 长任务目前主要通过线程执行，任务状态保存在内存中，难以扩展到多进程、多实例或 Worker。
- 行情、K线、市场快照等接口容易出现重复请求，用户多时会放大第三方 API 压力。
- 前端轮询频率偏高时，会增加后端压力；隐藏页面也可能继续请求。
- AI 选股、全市场筛选、回测等任务慢、贵、容易超时，需要更明确的并发控制和结果复用。

项目已有的有利基础：

- `requirements.txt` 已包含 `redis`、`sqlalchemy`、`asyncpg`、`alembic` 等依赖。
- `backend/app/core/redis.py` 已有 Redis 连接雏形。
- `backend/app/core/database.py` 已有 SQLAlchemy 异步数据库连接雏形。
- `backend/app/models/` 已存在用户、股票、行情、模拟盘、推荐、自选股等模型基础。
- `ai_stock_picker.py` 已经有分批 AI 评分和复核池方向，架构思路是对的。

## 3. 十个优化点

### 3.1 给行情、K线、市场快照、新闻、板块、AI 结果加缓存

目标是减少重复请求，尤其是避免前端用户请求直接放大第三方行情 API 压力。

建议缓存策略：

| 数据类型 | 建议缓存时间 | 说明 |
| --- | --- | --- |
| 实时行情 | 3-10 秒 | 交易时段短缓存，非交易时段可延长 |
| 市场快照 | 10-30 秒 | 首页、列表、详情页可复用 |
| 日 K | 1-10 分钟，非交易时段更久 | 不应每次打开详情都重新拉取 |
| 分钟 K / 分时 | 3-30 秒 | 热门股票短缓存，普通股票可稍长 |
| 新闻 | 5-30 分钟 | 新闻刷新失败时回退旧缓存 |
| 板块数据 | 30-60 秒 | 板块热度、资金流、排名可复用 |
| AI 选股结果 | 5-30 分钟 | 按策略、范围、参数生成缓存 key |

推荐数据流：

```text
第三方行情 API
  ↓
后端定时/按需拉取
  ↓
Redis / 本地缓存
  ↓
FastAPI 接口返回缓存结果
  ↓
前端展示
```

阶段 1 可以先使用内存和本地文件缓存，阶段 2 再统一迁入 Redis。

### 3.2 限制 AI 选股、全市场筛选、回测任务并发

AI 选股、全市场筛选、回测、日报周报都属于重任务，不能无限启动。

建议策略：

- 同一类任务设置最大并发数。
- 同一用户、同一策略、同一参数短时间内复用已有任务或缓存结果。
- 如果已有任务正在运行，接口返回当前 job_id，而不是重复启动。
- AI 批次失败允许有限次数重试。
- 超过并发上限时返回明确提示：任务繁忙，请稍后重试或查看已有任务。

### 3.3 降低前端轮询频率

当前长任务通过轮询获取进度，这个方式可以保留，但需要降频和停止条件。

建议策略：

```text
任务刚开始：1 秒一次
任务中期：2-3 秒一次
任务接近完成：1 秒一次
任务完成/失败：立即停止轮询
页面切到后台：降频或暂停
```

前端可使用：

```javascript
document.visibilityState
```

当页面不在前台时，减少请求频率。后续阶段可以再评估 SSE 或 WebSocket 推送。

### 3.4 job 状态从 main_simple.py 内存变量迁到 Redis

当前 `_ai_pick_jobs`、`_screening_jobs` 等内存 job 状态只在当前进程有效。

迁移到 Redis 后可以解决：

- 服务重启后状态更容易恢复或过期清理。
- 多进程、多实例可以共享任务状态。
- Worker 和 FastAPI 可以通过 Redis 交换任务进度。
- 前端轮询接口不依赖某个 Python 进程内存。

Redis key 建议：

```text
job:ai_pick:{job_id}
job:screening:{job_id}
job:backtest:{job_id}
job:report:{job_id}
```

job 状态建议保留 TTL，避免历史任务无限堆积。

### 3.5 用户配置、模拟持仓、订单、策略配置迁到 PostgreSQL

PostgreSQL 负责核心业务持久化。

优先迁移：

- 用户配置
- 自选股
- 模拟持仓
- 模拟订单
- 策略配置
- AI 推荐历史
- 风控记录
- 任务历史
- 策略记忆
- 模型配置

迁移原则：

- 第一阶段不强制迁移，先保留本地 JSON 和内存逻辑。
- 第二阶段接入 Redis 后，开始设计数据库表和迁移脚本。
- 第三阶段逐步切读写路径，旧文件作为导入源或兼容回退。
- 涉及 API Key 的字段必须继续脱敏返回，不能明文暴露给前端。

### 3.6 拆分 main_simple.py 为多个 API router

`main_simple.py` 目前承担太多职责，应逐步拆成 router。

建议拆分方向：

```text
auth_router.py
stocks_router.py
market_router.py
screening_router.py
signals_router.py
ai_router.py
portfolio_router.py
risk_router.py
sectors_router.py
settings_router.py
backtest_router.py
admin_router.py
```

拆分原则：

- 先迁接口定义，不大改业务逻辑。
- 业务仍调用现有 service。
- 每次只拆一个领域，保证可回滚。
- 拆分后 `main_simple.py` 只保留 app 初始化、中间件、静态页面、router 注册和后台循环启动。

### 3.7 拆分 index.html 内联 JS

短期不必迁移 React/Vue。先把原生 H5 拆文件即可明显降低维护难度。

保守拆分路线：

```text
backend/static/js/api.js
backend/static/js/state.js
backend/static/js/stocks.js
backend/static/js/aiPicker.js
backend/static/js/screener.js
backend/static/js/portfolio.js
backend/static/js/charts.js
backend/static/js/ui.js
```

拆分原则：

- 先抽纯工具：API 请求、格式化、图表配置。
- 再抽业务模块：AI 选股、智能筛选、持仓、行情。
- 保持页面行为不变。
- 不在第一阶段引入构建工具。

### 3.8 统一日志、错误处理、请求耗时统计

后端应统一记录：

- 请求路径
- HTTP 方法
- 状态码
- 耗时
- 用户标识或匿名会话标识
- 错误摘要
- job_id
- 任务类型

统一错误响应建议：

```json
{
  "ok": false,
  "error": "用户可读错误",
  "error_code": "AI_TIMEOUT",
  "request_id": "..."
}
```

这样前端可以稳定展示错误，不需要到处解析不同格式。

### 3.9 慢接口性能记录

先记录真实瓶颈，再决定优化顺序。

建议记录：

- 超过 1 秒的接口。
- 超过 5 秒的接口。
- AI 调用耗时。
- 第三方行情接口耗时。
- K线读取/计算耗时。
- 全市场筛选耗时。
- 回测耗时。

慢接口日志可以先写普通日志，后续再进入 PostgreSQL 或监控系统。

### 3.10 AI、筛选、回测迁到独立 Worker

Worker 是阶段 4 的目标，不建议第一阶段立刻做。

最终形态：

```text
FastAPI：接收请求、鉴权、查询状态、返回结果
  ↓
Redis：任务队列、任务状态、短期缓存
  ↓
Worker：执行 AI 选股、全市场筛选、回测、日报周报
  ↓
PostgreSQL：保存最终业务结果
```

迁移顺序：

1. 先把 job 状态标准化。
2. 再把 job 状态迁 Redis。
3. 再把任务启动逻辑抽象成统一接口。
4. 最后引入 Worker 消费任务。

## 4. 分阶段实施路线

### 阶段 1：小改稳态

目标：少改代码，立刻提升稳定性。

保留：

- 一台服务器
- FastAPI 单体
- 原生 H5
- 内存 job
- 本地缓存

实施内容：

1. 给热点接口加短缓存。
2. 给 AI 选股结果加短期缓存。
3. 限制 AI 选股、全市场筛选、回测并发。
4. 降低前端轮询频率。
5. 完成/失败后立即停止轮询。
6. 页面隐藏时降低请求频率。
7. 添加统一错误响应。
8. 添加请求耗时日志。
9. 记录慢接口。
10. 整理缓存 TTL 常量。

阶段 1 验收标准：

- 热门行情接口重复请求明显减少。
- 同条件 AI 选股短时间内不会重复跑。
- 同类重任务不能无限并发启动。
- 前端后台页面请求频率下降。
- 慢接口日志能看到路径、耗时、状态码和错误。
- 现有功能行为不大变。

### 阶段 2：Redis

目标：解决内存状态不共享、轮询压力、缓存混乱问题。

实施内容：

1. 接入 Redis 健康检查。
2. job 状态迁 Redis。
3. 热点行情放 Redis。
4. 市场快照放 Redis。
5. AI 任务进度放 Redis。
6. 筛选结果摘要放 Redis。
7. 添加接口限流。
8. 为 SSE/WebSocket 做准备。

阶段 2 后架构：

```text
FastAPI
  ↓
Redis：缓存、任务状态、临时结果
  ↓
本地文件 / PostgreSQL
```

阶段 2 验收标准：

- 重启 FastAPI 后，未过期 job 状态仍可查询。
- 多个接口读取同一份 Redis 行情缓存。
- AI 任务进度不再依赖 `main_simple.py` 内存变量。
- Redis 不可用时有明确降级或错误提示。

### 阶段 3：PostgreSQL

目标：把核心业务数据持久化。

优先入库：

1. 用户配置
2. 自选股
3. 模拟持仓
4. 模拟订单
5. 回测记录
6. AI 推荐历史
7. 风控记录
8. 任务历史
9. 策略记忆
10. 模型配置

实施内容：

- 使用已有 SQLAlchemy 异步基础设施。
- 补齐缺失模型和迁移脚本。
- 先双写或导入，再切读路径。
- 本地 JSON 文件保留一段时间作为兼容回退。

阶段 3 验收标准：

- 服务重启后核心业务数据不丢失。
- 用户配置、持仓、订单以数据库为准。
- AI 推荐历史和任务历史可追溯。
- API Key 和模型配置仍脱敏返回。

### 阶段 4：任务队列 Worker

目标：重任务不影响主接口响应。

实施内容：

- 引入独立 Worker 进程。
- AI 选股、全市场筛选、回测、日报周报迁入 Worker。
- FastAPI 只负责创建任务、查询状态、读取结果。
- Redis 承担任务队列和进度状态。
- PostgreSQL 保存最终结果和历史记录。

阶段 4 验收标准：

- AI 选股运行时，健康检查和普通行情接口仍快速响应。
- Worker 异常不会拖垮 FastAPI。
- 失败任务可查看错误原因。
- 同类任务有并发上限和排队策略。

### 阶段 5：前端工程化

目标：降低页面维护成本和渲染压力。

保守路线：

- 继续原生 H5。
- 拆 JS 文件。
- 图表按需加载。
- 隐藏 tab 不立即渲染。
- 股票列表、筛选结果、AI 推荐结果分页或虚拟列表。

正规路线：

- 后续评估 Vue / React / Next.js。
- 由于项目已有 `miniapp` 是 uni-app/Vue 风格，可以优先评估 Vue 路线。
- 但第一阶段不引入前端大框架。

阶段 5 验收标准：

- `index.html` 明显变薄。
- 页面首次加载更快。
- 几千条股票数据不会一次性全部渲染。
- 切换 tab 不会触发大量无效请求和图表渲染。

## 5. AI 选股优化策略

AI 选股最容易慢、贵、超时，应继续坚持“本地规则先筛，AI 只复核候选池”的方向。

推荐流程：

```text
全市场股票
  ↓
本地规则粗筛
  ↓
得到 100-300 只候选
  ↓
AI 分批评分
  ↓
二次复核
  ↓
最终推荐/观察 10-30 只
  ↓
结果入库/缓存
```

关键控制：

- 不把几千只股票一次性丢给 AI。
- 批次大小可配置。
- 并发批次数可配置。
- 失败批次有限重试。
- 相同策略、范围、参数短时间内复用结果。
- 质量评分和最终复核拆成两个阶段。
- AI 超时要返回明确错误或复用最近成功结果，不能静默空白。

缓存 key 建议包含：

```text
策略类型
选股范围
用户关键参数
交易日期
行情缓存版本
模型配置版本
```

## 6. 行情与 K 线缓存策略

行情接口应从“用户触发拉取”逐步变成“后端统一拉取、统一缓存、统一返回”。

推荐流程：

```text
第三方行情 API
  ↓
后端定时拉取 / 热点股票预热
  ↓
Redis 缓存
  ↓
FastAPI 接口读取缓存
  ↓
前端展示
```

交易时段策略：

- 热门股票实时行情 3-10 秒刷新。
- 自选股、持仓股、AI 推荐股优先预热。
- 普通股票按需读取，短时间内复用缓存。
- 分时数据短缓存。

非交易时段策略：

- 实时行情 TTL 可延长。
- 日 K 可长缓存。
- 新闻和板块可以低频刷新。
- 避免无意义高频轮询。

## 7. 前端性能策略

大文件本身不是最大性能问题，一次性渲染太多 DOM 才是常见卡顿来源。

优先优化：

- 股票列表分页。
- 筛选结果分页。
- AI 推荐和复核池分页。
- 大列表使用虚拟列表。
- 图表按需加载。
- 隐藏 tab 不立即渲染。
- 页面后台时暂停或降低轮询。
- 任务完成或失败后停止轮询。

## 8. 风险与注意事项

- 不要一次性把 Redis、PostgreSQL、Worker、router 拆分、前端模块化全部同时做，会让问题难以定位。
- Redis 不应替代数据库保存核心业务数据。
- PostgreSQL 不应承担高频实时行情热缓存。
- Worker 引入前必须先统一 job 状态结构。
- API Key、模型配置迁移时必须保留脱敏逻辑。
- 行情缓存必须区分交易时段和非交易时段。
- AI 结果缓存必须包含关键参数，避免用户改策略后误用旧结果。
- 拆 router 时要先保持接口路径和返回结构不变。
- 拆前端 JS 时要先保持页面行为不变，不同时做 UI 大改。

## 9. 后续实施建议

推荐下一步不是直接上 Redis 或 PostgreSQL，而是先做阶段 1：

1. 添加统一请求耗时日志和慢接口记录。
2. 给行情、市场快照、K线、板块、新闻接口梳理 TTL。
3. 给 AI 选股结果加短期缓存。
4. 给 AI 选股、筛选、回测加并发上限。
5. 降低前端轮询频率，并在页面隐藏时降频。

完成阶段 1 后，再进入 Redis 阶段。这样每一步都有明确收益，也不会把系统改成难以回滚的大工程。

## 10. 验收清单

文档阶段验收：

- 已覆盖当前架构现状。
- 已覆盖十个优化点。
- 已覆盖五阶段路线。
- 已覆盖缓存、AI、轮询、Redis、PostgreSQL、Worker、router 拆分、前端拆分策略。
- 没有要求立刻推翻现有 FastAPI 单体和原生 H5。

后续实现阶段验收：

- 热点行情接口重复请求明显减少。
- AI 选股相同条件短时间内复用结果。
- 同时启动多个重任务时能被拒绝、排队或复用已有任务。
- 前端后台页面请求频率下降。
- 慢接口日志能看到路径、耗时、状态码和错误信息。
- 重启服务后，迁入 PostgreSQL 的核心业务数据不丢失。
- Redis 不可用时有清晰错误或降级策略。
- Worker 执行重任务时，主接口仍保持响应。
