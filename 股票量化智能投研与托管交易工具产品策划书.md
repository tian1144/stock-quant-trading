# 一、当前阶段重新定位

## 1.1 原策划调整原因

上一版策划偏完整平台化，包含多账户、多人协作、权限、收费、托管、自动交易、机构版等内容。当前实际需求是先做一个能快速落地、能让两三个人使用、能在模拟盘中实时验证效果的小程序。

因此当前阶段必须收缩范围，优先做最关键的闭环：

1. 获取股票数据。
2. 盘后筛选股票。
3. 按短线/长线逻辑生成推荐。
4. 小程序展示推荐结果。
5. 实时模拟盘下单。
6. 实时模拟持仓和盈亏。
7. 根据真实盘中行情进行模拟买卖。
8. 用模拟盘实操结果验收系统是否符合预期。

## 1.2 当前阶段产品定位

当前阶段产品定位为：

> 一个内部使用的股票量化智能选股与实时模拟交易小程序。

它不是大型 SaaS 平台，也不是机构交易系统，更不是实盘托管产品。

第一版只需要服务：

- 你本人。
- 另外 1 到 2 个测试人员。
- 少量内部模拟盘账户。
- 少量核心策略。

## 1.3 当前阶段核心原则

### 1.3.1 小而完整

系统不追求大而全，但必须形成完整闭环：

```text
数据采集 → 股票筛选 → 推荐评分 → 小程序展示 → 实时模拟买卖 → 持仓盈亏 → 复盘验收
```

### 1.3.2 先模拟，后实盘

第一版必须只做模拟盘。模拟盘符合预期后，才考虑实盘提醒或实盘接口。

### 1.3.3 先两三个人可用，再考虑规模化

当前不需要复杂租户、不需要企业权限、不需要收费系统。只需要简单登录、内部用户白名单和模拟账户。

### 1.3.4 保留升级接口

虽然当前不做大型平台，但代码结构要避免写死。后续应能平滑升级到：

- 多用户。
- 多账户。
- 实盘交易。
- 自动交易。
- 收益统计。
- 权限管理。
- 收费系统。

# 二、本阶段核心目标

## 2.1 产品目标

本阶段要做出来的小程序必须满足：

1. 能在手机上查看今日推荐股票。
2. 能区分短线推荐和长线推荐。
3. 能查看每只股票的推荐理由。
4. 能查看上涨概率、预估涨幅、建议买入价、建议卖出价、止损价。
5. 能把股票加入自选关注。
6. 能进行实时模拟买入。
7. 能进行实时模拟卖出。
8. 模拟盘价格要尽量接近真实行情实时变化。
9. 能看到模拟持仓、浮动盈亏、总资产、收益率。
10. 能记录每笔模拟交易。
11. 能盘后查看当天模拟交易复盘。
12. 能根据模拟结果判断系统是否符合预期。

## 2.2 技术目标

本阶段技术目标：

1. 后端稳定运行。
2. 行情数据能定时更新。
3. 模拟盘能实时读取最新价格。
4. 小程序页面响应速度快。
5. 数据库结构清晰。
6. 推荐算法可调整权重。
7. 后续能扩展到正式 Web 平台或实盘交易。

## 2.3 验收目标

你第一时间会用模拟盘实操验收，所以验收标准必须围绕模拟盘设计：

1. 推荐股票是否能按时生成。
2. 推荐理由是否清楚。
3. 建议买入价、止损价、卖出价是否合理。
4. 模拟买入是否按实时价格成交。
5. 模拟持仓盈亏是否随行情变化。
6. 模拟卖出是否准确计算收益。
7. 系统是否能记录交易全过程。
8. 模拟盘体验是否接近真实交易。

# 三、本阶段明确不做的内容

为了快速落地，当前阶段先不做以下内容：

## 3.1 不做复杂账户体系

暂时不做：

- 多账户管理。
- 机构账户。
- 托管账户。
- 账户权限分层。
- 资金归集。
- 用户分账。

当前只做：

- 简单用户登录。
- 每个用户一个默认模拟盘。
- 初始模拟资金可配置。

## 3.2 不做收费系统

暂时不做：

- 手续费分成。
- 收益分成。
- 订阅费。
- 账单系统。
- 支付系统。

当前只做：

- 模拟交易内部手续费计算。
- 例如模拟券商佣金、印花税、过户费。
- 不做平台收费。

## 3.3 不做真实实盘交易

暂时不接：

- 券商接口。
- 自动下真实订单。
- 真实资金管理。
- 实盘托管。

当前只做：

- 实时模拟盘。
- 买卖提醒。
- 模拟交易记录。

## 3.4 不做复杂 AI

暂时不做：

- AI 自动写策略。
- AI 自动交易。
- 大规模 RAG 投研库。

当前只做可选：

- 用规则生成推荐理由。
- 后期可接 AI 优化文字解释。

## 3.5 不做大型管理后台

暂时不做复杂后台。

当前只做一个简易管理功能：

- 手动触发数据更新。
- 查看定时任务状态。
- 修改评分权重。
- 修改模拟盘初始资金。

# 五、MVP 最终验收标准

## 5.1 数据验收

必须满足：

1. 股票基础列表完整。
2. 日线行情能正常更新。
3. 当日价格能实时或准实时更新。
4. 股票停牌、ST、退市风险能识别。
5. 推荐结果能每天生成。

## 5.2 推荐验收

必须满足：

1. 每天能输出短线推荐。
2. 每天能输出长线推荐。
3. 每只推荐股票有评分。
4. 每只推荐股票有推荐理由。
5. 每只推荐股票有买入价、卖出价、止损价。
6. 推荐结果能在小程序查看。

## 5.3 模拟盘验收

必须满足：

1. 可以模拟买入。
2. 可以模拟卖出。
3. 买入时检查现金是否足够。
4. 买入数量符合 A 股 100 股一手规则。
5. 卖出时检查可卖数量。
6. 支持 T+1 可卖限制。
7. 持仓盈亏随行情变化。
8. 总资产实时更新。
9. 交易记录准确保存。
10. 手续费、印花税可模拟计算。

## 5.4 体验验收

必须满足：

1. 小程序打开速度可接受。
2. 推荐列表清晰。
3. 模拟买卖流程简单。
4. 持仓盈亏直观。
5. 交易记录清楚。
6. 不需要复杂配置即可使用。

# 六、推荐最优技术选型

## 6.1 小程序前端

推荐使用：uni-app + Vue 3 + TypeScript。

原因：

- 可以开发微信小程序。
- 未来可扩展 H5、App。
- Vue 语法开发效率高。
- 适合小团队快速开发。

可选方案：

- 原生微信小程序：性能好，但后续跨端能力弱。
- Taro + React：适合 React 团队，但学习和工程复杂度略高。

本项目建议选择 uni-app。

## 6.2 后端

推荐使用：FastAPI + Python。

原因：

- Python 适合量化计算。
- FastAPI 开发接口快。
- 后续可接 AI、数据分析、机器学习。
- 小团队维护成本低。

## 6.3 数据库

当前阶段推荐：PostgreSQL + Redis。

PostgreSQL 用于：

- 用户。
- 股票基础数据。
- 日线行情。
- 推荐结果。
- 自选股。
- 模拟盘。
- 交易记录。

Redis 用于：

- 最新行情缓存。
- 实时价格快照。
- 推荐结果缓存。
- 小程序高频查询缓存。

暂时不引入 ClickHouse，等数据量变大后再升级。

## 6.4 定时任务

推荐：APScheduler。

原因：

- 当前只有 2 到 3 人使用。
- 任务量不大。
- 不需要一开始上 Celery。
- 部署简单。

后续升级：

- 数据采集任务变多后，再引入 Celery + Redis。

## 6.5 行情数据源

MVP 推荐：

- AkShare。
- Tushare。

第一版行情方案：

- 日线和基础数据用 Tushare / AkShare。
- 盘中实时价格可优先用 AkShare 可用接口或第三方行情接口。
- 如果实时行情不稳定，先做 5 秒到 30 秒刷新一次的准实时模拟盘。

注意：商业化前必须确认数据授权。

## 6.6 图表

小程序端图表建议：

- uCharts。
- ECharts 小程序版。

第一版可以先不做复杂 K 线，只做：

- 当前价。
- 涨跌幅。
- 简单分时。
- 最近 20 日走势。

# 七、当前阶段整体架构

## 7.1 MVP 架构图

```text
uni-app 小程序
  ↓ HTTPS API
FastAPI 后端
  ├─ 用户登录模块
  ├─ 股票数据模块
  ├─ 行情模块
  ├─ 推荐模块
  ├─ 自选股模块
  ├─ 模拟盘模块
  ├─ 提醒模块
  └─ 简易管理模块
  ↓
PostgreSQL 数据库
Redis 行情缓存
  ↑
APScheduler 定时任务
  ├─ 同步股票基础数据
  ├─ 同步日线行情
  ├─ 同步盘中价格
  ├─ 盘后筛选股票
  └─ 生成推荐结果
```

## 7.2 当前阶段服务模块

当前只需要以下模块：

1. auth：简单登录。
2. stock：股票基础信息。
3. market：行情数据。
4. recommendation：推荐结果。
5. watchlist：自选股。
6. simulator：模拟盘。
7. alert：提醒。
8. admin：简易管理。

## 7.3 当前阶段不拆微服务

后端保持一个 FastAPI 项目，内部模块化，不拆微服务。

原因：

- 只有 2 到 3 人使用。
- 业务还在验证阶段。
- 单体部署简单。
- 后续可按模块拆分。

## 15.6 模拟成交规则

第一版使用“准实时市价成交”：

- 买入时读取 Redis 最新价。
- 卖出时读取 Redis 最新价。
- 如果价格缺失，拒绝交易。
- 买入数量必须是 100 股整数倍。
- 买入金额不能超过可用现金。
- 卖出数量不能超过可卖数量。
- 当日买入股票不可当日卖出，符合 T+1。
- 停牌不可买卖。
- 涨停时可模拟买入失败或提示风险。
- 跌停时可模拟卖出失败或提示风险。

- 印花税：卖出成交额 0.05%。
- 过户费：可暂不计算，或按 0.001% 模拟。

# 二十六、总结

当前阶段最正确的做法是：

> 先做一个小而完整、能实时模拟交易、能让你亲自用模拟盘验收的小程序。

第一版不要做复杂账户、收费、实盘、托管、机构后台。核心只做：

1. 股票数据。
2. 推荐筛选。
3. 长短线推荐。
4. 小程序展示。
5. 实时模拟买卖。
6. 持仓盈亏。
7. 交易记录。
8. 模拟盘复盘。

只要模拟盘实操符合你的预期，后续再逐步升级到更完整的平台。这样开发成本最低、验证速度最快、风险最小，也最符合当前只有两三个人使用的实际情况。
