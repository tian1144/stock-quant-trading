# 一、当前阶段重新定位

## 1.1 原策划调整原因

上一版策划偏完整平台化，包含多账户、多人协作、权限、收费、托管、自动交易、机构版等内容。当前实际需求是先做一个能快速落地、能让两三个人使用、能在模拟盘中实时验证效果的小程序。

因此当前阶段必须收缩范围，优先做最关键的闭环：

1. 获取股票数据。
2. 盘后筛选股票。
3. 按短线/长线逻辑生成推荐。
4. 小程序展示推荐结果。
5. 实时模拟盘下单。
6. 实时模拟持仓和盈亏。
7. 根据真实盘中行情进行模拟买卖。
8. 用模拟盘实操结果验收系统是否符合预期。

## 1.2 当前阶段产品定位

当前阶段产品定位为：

> 一个内部使用的股票量化智能选股与实时模拟交易小程序。

它不是大型 SaaS 平台，也不是机构交易系统，更不是实盘托管产品。

第一版只需要服务：

- 你本人。
- 另外 1 到 2 个测试人员。
- 少量内部模拟盘账户。
- 少量核心策略。

## 1.3 当前阶段核心原则

### 1.3.1 小而完整

系统不追求大而全，但必须形成完整闭环：

```text
数据采集 → 股票筛选 → 推荐评分 → 小程序展示 → 实时模拟买卖 → 持仓盈亏 → 复盘验收
```

### 1.3.2 先模拟，后实盘

第一版必须只做模拟盘。模拟盘符合预期后，才考虑实盘提醒或实盘接口。

### 1.3.3 先两三个人可用，再考虑规模化

当前不需要复杂租户、不需要企业权限、不需要收费系统。只需要简单登录、内部用户白名单和模拟账户。

### 1.3.4 保留升级接口

虽然当前不做大型平台，但代码结构要避免写死。后续应能平滑升级到：

- 多用户。
- 多账户。
- 实盘交易。
- 自动交易。
- 收益统计。
- 权限管理。
- 收费系统。

# 二、本阶段核心目标

## 2.1 产品目标

本阶段要做出来的小程序必须满足：

1. 能在手机上查看今日推荐股票。
2. 能区分短线推荐和长线推荐。
3. 能查看每只股票的推荐理由。
4. 能查看上涨概率、预估涨幅、建议买入价、建议卖出价、止损价。
5. 能把股票加入自选关注。
6. 能进行实时模拟买入。
7. 能进行实时模拟卖出。
8. 模拟盘价格要尽量接近真实行情实时变化。
9. 能看到模拟持仓、浮动盈亏、总资产、收益率。
10. 能记录每笔模拟交易。
11. 能盘后查看当天模拟交易复盘。
12. 能根据模拟结果判断系统是否符合预期。

## 2.2 技术目标

本阶段技术目标：

1. 后端稳定运行。
2. 行情数据能定时更新。
3. 模拟盘能实时读取最新价格。
4. 小程序页面响应速度快。
5. 数据库结构清晰。
6. 推荐算法可调整权重。
7. 后续能扩展到正式 Web 平台或实盘交易。

## 2.3 验收目标

你第一时间会用模拟盘实操验收，所以验收标准必须围绕模拟盘设计：

1. 推荐股票是否能按时生成。
2. 推荐理由是否清楚。
3. 建议买入价、止损价、卖出价是否合理。
4. 模拟买入是否按实时价格成交。
5. 模拟持仓盈亏是否随行情变化。
6. 模拟卖出是否准确计算收益。
7. 系统是否能记录交易全过程。
8. 模拟盘体验是否接近真实交易。

# 三、本阶段明确不做的内容

为了快速落地，当前阶段先不做以下内容：

## 3.1 不做复杂账户体系

暂时不做：

- 多账户管理。
- 机构账户。
- 托管账户。
- 账户权限分层。
- 资金归集。
- 用户分账。

当前只做：

- 简单用户登录。
- 每个用户一个默认模拟盘。
- 初始模拟资金可配置。

## 3.2 不做收费系统

暂时不做：

- 手续费分成。
- 收益分成。
- 订阅费。
- 账单系统。
- 支付系统。

当前只做：

- 模拟交易内部手续费计算。
- 例如模拟券商佣金、印花税、过户费。
- 不做平台收费。

## 3.3 不做真实实盘交易

暂时不接：

- 券商接口。
- 自动下真实订单。
- 真实资金管理。
- 实盘托管。

当前只做：

- 实时模拟盘。
- 买卖提醒。
- 模拟交易记录。

## 3.4 不做复杂 AI

暂时不做：

- AI 自动写策略。
- AI 自动交易。
- 大规模 RAG 投研库。

当前只做可选：

- 用规则生成推荐理由。
- 后期可接 AI 优化文字解释。

## 3.5 不做大型管理后台

暂时不做复杂后台。

当前只做一个简易管理功能：

- 手动触发数据更新。
- 查看定时任务状态。
- 修改评分权重。
- 修改模拟盘初始资金。

# 五、MVP 最终验收标准

## 5.1 数据验收

必须满足：

1. 股票基础列表完整。
2. 日线行情能正常更新。
3. 当日价格能实时或准实时更新。
4. 股票停牌、ST、退市风险能识别。
5. 推荐结果能每天生成。

## 5.2 推荐验收

必须满足：

1. 每天能输出短线推荐。
2. 每天能输出长线推荐。
3. 每只推荐股票有评分。
4. 每只推荐股票有推荐理由。
5. 每只推荐股票有买入价、卖出价、止损价。
6. 推荐结果能在小程序查看。

## 5.3 模拟盘验收

必须满足：

1. 可以模拟买入。
2. 可以模拟卖出。
3. 买入时检查现金是否足够。
4. 买入数量符合 A 股 100 股一手规则。
5. 卖出时检查可卖数量。
6. 支持 T+1 可卖限制。
7. 持仓盈亏随行情变化。
8. 总资产实时更新。
9. 交易记录准确保存。
10. 手续费、印花税可模拟计算。

## 5.4 体验验收

必须满足：

1. 小程序打开速度可接受。
2. 推荐列表清晰。
3. 模拟买卖流程简单。
4. 持仓盈亏直观。
5. 交易记录清楚。
6. 不需要复杂配置即可使用。

# 六、推荐最优技术选型

## 6.1 小程序前端

推荐使用：uni-app + Vue 3 + TypeScript。

原因：

- 可以开发微信小程序。
- 未来可扩展 H5、App。
- Vue 语法开发效率高。
- 适合小团队快速开发。

可选方案：

- 原生微信小程序：性能好，但后续跨端能力弱。
- Taro + React：适合 React 团队，但学习和工程复杂度略高。

本项目建议选择 uni-app。

## 6.2 后端

推荐使用：FastAPI + Python。

原因：

- Python 适合量化计算。
- FastAPI 开发接口快。
- 后续可接 AI、数据分析、机器学习。
- 小团队维护成本低。

## 6.3 数据库

当前阶段推荐：PostgreSQL + Redis。

当前实际落地状态：

- PostgreSQL：已有 `SQLAlchemy + asyncpg` 连接配置、ORM 模型和依赖，但主运行入口 `backend/app/main_simple.py` 暂未走数据库读写，也没有正式 Alembic 迁移目录；现阶段属于“已预留、未生产化承载核心业务”。
- Redis：已有连接封装和行情快照 Key 设计，但当前核心行情、AI 结果、模拟盘、任务状态主要仍由内存状态与 `backend/data/**` 下的 JSON/cache 文件承载；Redis 属于“已预留、未成为主缓存层”。
- 向量库：当前没有接入 pgvector、FAISS、Milvus、Qdrant、Chroma 等生产向量库；游资资料、PDF/OCR 精华和策略记忆以结构化 JSON 学习笔记保存。
- RAG：当前不是完整 RAG 检索系统，而是“策略记忆注入”模式：后端会把最近学习笔记摘要加入 AI 上下文，让不同模型共享站内经验；还没有做 embedding、相似度检索、召回重排和引用溯源。

也就是说，当前更接近“JSON/cache 文件 + 内存状态 + 本地任务日志”的单机增强版。它适合快速迭代、方便排查、单人或小范围使用；但如果要生产化、多用户、高并发、长期审计和故障恢复，就需要把核心状态迁入 PostgreSQL，把高频短生命周期数据迁入 Redis，把投研资料迁入向量库/RAG。

PostgreSQL 用于：

- 用户。
- 股票基础数据。
- 日线行情。
- 推荐结果。
- 自选股。
- 模拟盘。
- 交易记录。

Redis 用于：

- 最新行情缓存。
- 实时价格快照。
- 推荐结果缓存。
- 小程序高频查询缓存。

向量库 / RAG 用于：

- 保存公告、研报、课程资料、交割单 OCR、AI 复盘笔记等非结构化投研知识。
- 按股票、题材、情绪周期、游资席位、K线结构、历史案例相似度召回材料。
- 给 AI 选股、风控复核、复盘总结提供“可追溯依据”，减少只靠当前提示词和短期上下文的漂移。
- 把长期经验从“最近几条记忆注入”升级为“按问题动态召回证据”。

暂时不引入 ClickHouse，等数据量变大后再升级。

## 6.4 定时任务

推荐：APScheduler。

原因：

- 当前只有 2 到 3 人使用。
- 任务量不大。
- 不需要一开始上 Celery。
- 部署简单。

后续升级：

- 数据采集任务变多后，再引入 Celery + Redis。

## 6.5 行情数据源

MVP 推荐：

- AkShare。
- Tushare。

第一版行情方案：

- 日线和基础数据用 Tushare / AkShare。
- 盘中实时价格可优先用 AkShare 可用接口或第三方行情接口。
- 如果实时行情不稳定，先做 5 秒到 30 秒刷新一次的准实时模拟盘。

注意：商业化前必须确认数据授权。

## 6.6 图表

小程序端图表建议：

- uCharts。
- ECharts 小程序版。

第一版可以先不做复杂 K 线，只做：

- 当前价。
- 涨跌幅。
- 简单分时。
- 最近 20 日走势。

# 七、当前阶段整体架构

## 7.1 MVP 架构图

```text
uni-app 小程序
  ↓ HTTPS API
FastAPI 后端
  ├─ 用户登录模块
  ├─ 股票数据模块
  ├─ 行情模块
  ├─ 推荐模块
  ├─ 自选股模块
  ├─ 模拟盘模块
  ├─ 提醒模块
  └─ 简易管理模块
  ↓
PostgreSQL 数据库
Redis 行情缓存
  ↑
APScheduler 定时任务
  ├─ 同步股票基础数据
  ├─ 同步日线行情
  ├─ 同步盘中价格
  ├─ 盘后筛选股票
  └─ 生成推荐结果
```

## 7.2 当前阶段服务模块

当前只需要以下模块：

1. auth：简单登录。
2. stock：股票基础信息。
3. market：行情数据。
4. recommendation：推荐结果。
5. watchlist：自选股。
6. simulator：模拟盘。
7. alert：提醒。
8. admin：简易管理。

## 7.3 当前阶段不拆微服务

后端保持一个 FastAPI 项目，内部模块化，不拆微服务。

原因：

- 只有 2 到 3 人使用。
- 业务还在验证阶段。
- 单体部署简单。
- 后续可按模块拆分。

# 十九、数据库设计汇总

## 19.0 当前数据底座状态与作用

当前项目的数据底座分为四类：

1. 业务主状态：目前主要在 `backend/data`、`backend/portfolio_state.json`、`backend/data/ai/strategy_memory.json`、任务结果 JSON 和内存 `state_store` 中。
2. 行情缓存：目前主要在 `backend/data/market_cache`、`news_cache`、`sector_cache`、`disclosures` 等本地缓存目录中。
3. Agent/任务审计：目前主要在 `backend/data/agents/{agent_id}/state.json`、`events.jsonl` 和 artifacts 中。
4. AI 知识记忆：目前主要在策略记忆 JSON、游资资料抽取报告、PDF/OCR 报告中。

这些 JSON/cache 文件的作用：

- 快速落地：不依赖外部数据库服务，开发和本机验证速度快。
- 可读可查：出错时能直接打开文件看状态、任务结果和 AI 输出。
- 便于迭代：AI 选股、公告缓存、OCR 学习、策略记忆仍在高频调整，用文件存储更轻。
- 低成本恢复：本地服务重启后可以从文件恢复股票池、模拟盘、AI 推荐和策略记忆。

它的边界：

- 不适合多用户并发写入。
- 缺少事务、唯一约束、行级锁和完整迁移管理。
- 文件越来越多后，查询、分页、聚合和审计会变慢。
- 不能替代 Redis 的高频 TTL 缓存，也不能替代向量库的语义检索。

生产化落地顺序：

1. PostgreSQL 先承载强一致业务状态：用户、自选、模拟盘、订单、推荐、选股任务、AI 复核结果、公告风险索引。
2. Redis 承载短生命周期高频数据：最新价、盘口快照、任务进度、按钮防重复、热点榜单缓存、AI 批次临时状态。
3. 向量库承载知识检索：游资交割单、OCR 课程截图、公告文本、研报摘要、复盘报告、历史案例。
4. RAG 服务把“向量召回 + 结构化因子 + 策略记忆”组合进 AI 上下文，并返回引用来源、命中片段和置信度。
5. JSON/cache 文件降级为冷备、导出、调试、审计归档，不再作为生产主存储。

## 19.1 生产化状态矩阵

| 模块 | 当前状态 | 已有基础 | 缺口 | 生产化目标 |
| --- | --- | --- | --- | --- |
| PostgreSQL | 预留，未成为主链路 | `requirements.txt` 已含 `sqlalchemy`、`asyncpg`、`psycopg2-binary`；已有 `app/core/database.py` 和 ORM 模型 | 没有 Alembic 迁移；`main_simple.py` 未接入数据库仓储；核心状态仍在 JSON | 承载强一致业务数据、任务结果、模拟盘和审计索引 |
| Redis | 预留，未成为主缓存 | 已有 `app/core/redis.py`，设计了 `market:snapshot:{code}` | 当前行情和任务状态仍主要在文件和内存；未统一 TTL、Key 命名、降级策略 | 承载高频行情快照、任务进度、临时锁和热点缓存 |
| 向量库 | 未接入 | 已有游资资料抽取、PDF/OCR 报告、策略记忆 JSON | 没有 embedding、chunk、向量索引、相似度召回 | 承载非结构化投研资料的语义检索 |
| RAG | 策略记忆注入阶段 | `strategy_memory_service` 会把最近学习笔记加入 AI 上下文 | 没有按问题召回、重排、引用溯源、知识版本 | 给 AI 选股/风控/复盘动态提供证据 |
| JSON/cache | 当前主承载 | `backend/data/**`、任务 JSON、Agent `state.json/events.jsonl` | 并发、事务、查询和审计能力不足 | 转为冷备、调试、导出和审计归档 |

这个矩阵是当前项目的事实口径：不能把 PostgreSQL、Redis、向量库、RAG 写成已经生产化；它们是下一阶段生产化改造的目标。

## 19.2 PostgreSQL 生产化设计

PostgreSQL 的作用是保存必须可靠、可查询、可审计、可恢复的业务数据。凡是涉及用户资产、模拟盘、订单、推荐结论、风控结果、任务状态和长期历史的数据，都应该进入 PostgreSQL。

优先落库对象：

1. 用户与配置：`users`、`user_settings`、`system_configs`、`ai_model_configs`。
2. 股票主数据：`stocks`、`stock_universe_snapshots`、`stock_universe_changes`。
3. 行情历史：`stock_daily_bars`、`stock_intraday_bars`、`money_flow_bars`、`chip_distributions`。
4. 选股与 AI 结果：`screening_jobs`、`screening_candidates`、`ai_pick_jobs`、`ai_quality_scores`、`ai_reviewed_candidates`、`recommendations`。
5. 风控与公告：`disclosure_risks`、`risk_reviews`、`risk_events`。
6. 模拟盘：`sim_portfolios`、`sim_positions`、`sim_orders`、`sim_trades`、`portfolio_daily_snapshots`。
7. Agent 审计：`agent_tasks`、`agent_events`、`agent_artifacts_index`。
8. 策略学习：`strategy_memory_notes`、`trade_review_reports`、`factor_weight_change_requests`。

关键约束：

- 股票代码、交易日、任务 ID、订单 ID 必须有唯一约束。
- 订单、持仓、现金变化必须在同一事务内完成。
- AI 推荐和风控复核必须带 `model_name`、`prompt_version`、`input_hash`、`output_hash`。
- 历史行情数据只追加或幂等覆盖，不允许静默删除。
- 模拟盘资金、持仓和订单必须支持重建，即可从成交流水重新算出当前状态。

第一阶段可以先不用复杂分库分表，但必须建立迁移目录和版本号：

```text
backend/alembic/
backend/alembic/versions/
backend/alembic.ini
```

验收标准：

- 停掉服务再启动，模拟盘、AI 推荐、任务状态不依赖 JSON 也能恢复。
- 同一只股票同一交易日不会重复插入多条日K。
- 同一个后台任务不会因为刷新页面重复创建结果。
- 任意推荐都能通过数据库查到候选来源、AI 打分、风控复核和模拟信号。

## 19.3 Redis 生产化设计

Redis 的作用不是替代数据库，而是承载短生命周期、高频访问、允许过期重建的数据。

推荐 Key 规范：

```text
market:snapshot:{code}              最新行情快照，TTL 5-30 秒
market:orderbook:{code}             买卖盘口，TTL 5-30 秒
market:sector:hot                   热门板块榜，TTL 30-120 秒
screening:job:{job_id}:progress     智能筛选任务进度，TTL 24 小时
ai_pick:job:{job_id}:progress       AI 选股任务进度，TTL 24 小时
ai_quality:batch:{job_id}:{idx}     AI 批次临时结果，TTL 24 小时
lock:screening:{scope}:{strategy}   防重复运行锁，TTL 按任务超时设置
lock:order:{portfolio_id}:{code}    模拟下单防重复锁，TTL 10-60 秒
cache:stock_detail:{code}           股票详情聚合缓存，TTL 30-300 秒
cache:disclosure_risk:{code}        公告风险聚合缓存，TTL 1-24 小时
```

使用原则：

- Redis 里只放可重建数据，不能只把唯一真实状态放 Redis。
- 关键任务完成后，最终结果必须写 PostgreSQL；Redis 只保留进度和临时状态。
- Redis 失败时，系统可以降级到 PostgreSQL/JSON 读取，但需要在健康检查中标记。
- 所有锁都必须设置 TTL，避免异常中断后永久锁死。

验收标准：

- 页面刷新时能从 Redis 快速读到任务进度。
- Redis 清空后，系统不会丢失最终推荐、订单和策略记忆。
- 行情快照过期后能自动从数据源或本地缓存重建。

## 19.4 向量库生产化设计

向量库的作用是保存非结构化投研知识，并按语义相似度召回证据。它适合保存“文字很多、结构松散、需要按问题检索”的资料，不适合保存订单、持仓、资金和强一致业务状态。

建议优先采用两种路线之一：

1. 简化路线：PostgreSQL + pgvector。优点是部署简单，数据和向量在同一个数据库里，适合当前小团队。
2. 独立路线：Qdrant 或 Milvus。优点是向量检索能力更完整，适合资料库变大后独立扩容。

当前项目建议先走 pgvector，因为它和 PostgreSQL 生产化可以共用部署、备份、权限和迁移。

知识库分层：

- `trade_notes`：游资交割单、短线课程、PDF/OCR 文本。
- `disclosures`：公告、财报、监管函、减持、业绩修正。
- `research_notes`：研报摘要、板块逻辑、政策事件。
- `review_reports`：模拟盘复盘、亏损归因、成功样本。
- `strategy_memory`：人工沉淀和 AI 总结的策略习惯。
- `historical_cases`：历史相似形态、回测样本、典型交易案例。

每个 chunk 至少保留：

- `doc_id`
- `chunk_id`
- `source_type`
- `source_path`
- `title`
- `stock_code`
- `sector`
- `trade_date`
- `author_or_origin`
- `text`
- `embedding`
- `confidence`
- `created_at`
- `version`

验收标准：

- 输入“低位回落到箱体支撑但公告有减持风险”时，能召回相关游资经验、公告规则和历史案例。
- AI 输出推荐理由时能列出引用来源，而不是只写泛泛判断。
- OCR 低置信度内容不能直接参与权重训练，只能作为备注或弱证据。

## 19.5 RAG 生产化设计

RAG 的作用是把“结构化因子”和“非结构化知识证据”合并给 AI。它不是让 AI 自由搜索所有资料，而是按任务类型受控召回、压缩、引用和审计。

推荐流程：

```text
用户/任务问题
  -> 解析任务类型：选股、风控、复盘、公告解释、策略学习
  -> 提取股票代码、板块、日期、形态、风险标签
  -> 结构化查询 PostgreSQL
  -> 语义查询向量库
  -> 按时间、来源可信度、相关性重排
  -> 压缩为 AI 上下文
  -> AI 输出结论、引用、置信度
  -> 保存输入、召回证据、输出和模型元数据
```

不同任务的召回策略：

| 任务 | 结构化输入 | 向量召回内容 | 输出要求 |
| --- | --- | --- | --- |
| AI 选股 | 分项评分、K线、资金、历史有效性、游资经验因子 | 相似交易案例、策略记忆、板块经验 | 推荐/观察/否决，必须说明证据 |
| 风控复核 | 公告风险、仓位、黑名单、模型输出 | 暴雷案例、公告规则、亏损复盘 | 通过/降级/否决，允许一票否决 |
| 复盘学习 | 模拟订单、盈亏、持仓周期 | 历史相似交易、游资纪律、失败案例 | 学习笔记和权重建议，不直接改生产权重 |
| 公告解释 | 公告标题、正文、风险标签 | 历史公告影响、监管案例 | 风险级别、影响路径、是否进入观察 |

RAG 安全边界：

- 召回证据必须带来源和时间。
- 低置信 OCR 只能作为弱证据。
- AI 不得把召回资料当作实时行情。
- RAG 只能辅助判断，最终交易仍必须经过风控。
- 每次召回都要记录 `query_hash`、`retrieved_doc_ids`、`prompt_version`。

验收标准：

- 同一个问题重复查询，召回结果稳定可解释。
- AI 最终结论能追溯到结构化数据和知识片段。
- 资料更新后有版本号，旧推荐能还原当时使用的知识版本。

## 19.6 JSON/cache 到生产底座迁移计划

迁移必须渐进，不建议一次性替换全部文件存储。

Phase 1：只读双写校验

- 保持现有 JSON/cache 读写不变。
- 新增 PostgreSQL 表和 Redis Key。
- 关键状态写 JSON 的同时双写 PostgreSQL。
- 后台比对 JSON 和 PostgreSQL 的记录数量、hash 和关键字段。

Phase 2：主读切换

- 低风险模块先从 PostgreSQL 读取，例如自选、推荐历史、任务状态。
- JSON 保留为导出和回滚备份。
- Redis 接管任务进度和行情聚合缓存。

Phase 3：强一致模块迁移

- 模拟盘订单、持仓、现金流水切到 PostgreSQL 事务。
- 下单和撤单加 Redis 短锁，防止重复点击。
- AI 推荐、风控复核、模拟信号统一写入数据库并关联 `task_id`。

Phase 4：RAG 接入

- 把游资资料、PDF/OCR、公告、复盘报告切 chunk。
- 生成 embedding 并写入 pgvector 或独立向量库。
- AI 选股和风控复核开始按任务召回证据。

Phase 5：JSON 降级

- `backend/data/**` 只保留冷备、导出、审计归档和调试 artifact。
- 生产查询不再依赖散落 JSON 文件。
- 所有定时任务和后台任务使用统一任务表恢复。

## 19.7 健康检查与验收面板

生产化后需要新增一个数据底座健康面板，建议接口：

```text
GET /api/v1/system/storage/status
GET /api/v1/system/storage/coverage
GET /api/v1/system/rag/status
GET /api/v1/system/cache/status
```

面板展示：

- PostgreSQL 是否可连接、迁移版本、核心表行数。
- Redis 是否可连接、关键 Key 数量、命中率、锁数量。
- 向量库是否可连接、文档数、chunk 数、embedding 维度、最近更新时间。
- RAG 是否启用、最近召回耗时、召回命中率、低置信 OCR 占比。
- JSON/cache 与数据库的双写校验状态。
- 最近 20 个任务是否能从任务表恢复。

最终验收：

- 关闭 Redis 不丢最终业务结果，只影响实时速度。
- 清空 JSON/cache 后，核心用户状态、模拟盘、推荐历史、任务结果仍可从 PostgreSQL 恢复。
- 新增一份投研资料后，RAG 能召回并在 AI 输出中给出来源。
- 任一 AI 推荐都能串起：股票数据 -> 量化评分 -> RAG 证据 -> AI 结论 -> 风控复核 -> 模拟信号。

## 19.8 AI 记忆与自主学习验收规则

AI 记忆是本项目的一级核心能力，优先级高于单次模型输出效果。系统必须保证：更换主模型、更换独立风控模型、切换 OpenAI 兼容网关、AI 托管复盘、人工追加学习笔记之后，站内策略记忆都能继续注入后续 AI 上下文。

必须长期保留的能力：

- 模型无关：记忆保存在站内，不绑定 DeepSeek、OpenAI、Claude、Gemini 或其他兼容网关。
- 自动注入：`chat_json`、`chat_text`、独立风控模型调用都必须注入 `strategy_memory_service.get_model_memory_context()`。
- 自主学习：AI 托管成交、模拟盘复盘、人工校验交割单、OCR/PDF/视频学习笔记都必须能写入策略记忆。
- 反思边界：单笔交易反思只能作为学习笔记，不允许未经足够样本和回测直接改生产权重。
- 可测试：任何改动 AI 调用、模型配置、策略记忆、RAG 或风控复核代码时，必须跑 AI 记忆回归测试。

当前已建立的回归测试：

```text
backend/tests/test_strategy_memory_integration.py
```

覆盖范围：

- 学习笔记追加后，记忆上下文能包含最新反思。
- 主模型 `chat_json` 调用会收到站内策略记忆和记忆版本。
- 切换主模型后，`chat_text` 仍能收到同一份记忆。
- 独立风控模型也会收到同一份策略记忆。

验收命令：

```text
backend\venv\Scripts\python.exe -m unittest backend.tests.test_strategy_memory_integration -v
```

通过标准：

- 测试必须离线运行，不依赖真实外部模型。
- 测试必须能捕获模型请求体，确认 system prompt 中存在策略记忆。
- 测试必须验证换模型后记忆不丢失。
- 测试必须验证独立风控模型不绕过记忆。

### 19.8.1 AI 记忆权限与所有权

AI 记忆属于系统最高敏感资产，权限必须比普通业务数据更严格。后续账户系统上线后，AI 记忆不能按普通用户资料处理，必须单独建权限模型。

权限原则：

- 普通账号：无权查看完整 AI 记忆；只能看到 AI 输出给自己的推荐、解释和公开摘要。
- 管理账号：最多拥有只读查看权限，可查看记忆版本、摘要、来源、更新时间和审计记录，但不能修改、删除或重建记忆。
- 终端管理员：唯一拥有 AI 记忆的新增、修改、删除、恢复、重建、导入、导出权限。当前终端管理员就是我本人。
- AI 自主学习：只能追加候选学习笔记，不能直接覆盖、删除或改写既有记忆；所有自主学习必须进入待审计记录。
- 风控模型和主模型：只能读取注入后的上下文片段，不能直接访问完整记忆文件或记忆管理接口。

后续账户系统必须预留角色：

```text
user                 普通用户
readonly_admin       只读管理账号
terminal_admin       终端管理员，唯一记忆写权限
system_agent         系统 Agent，仅可追加待审核学习记录
risk_auditor         风控审计账号，只读查看风险相关记忆摘要
```

AI 记忆相关接口权限：

| 接口能力 | 普通账号 | 管理账号 | 终端管理员 | 系统 Agent |
| --- | --- | --- | --- | --- |
| 查看记忆摘要 | 禁止 | 允许 | 允许 | 禁止 |
| 查看完整记忆 | 禁止 | 只读允许 | 允许 | 禁止 |
| 追加人工学习笔记 | 禁止 | 禁止 | 允许 | 禁止 |
| 追加 AI 自主反思候选 | 禁止 | 禁止 | 允许 | 允许，但必须标记为待审核 |
| 修改记忆 | 禁止 | 禁止 | 允许 | 禁止 |
| 删除记忆 | 禁止 | 禁止 | 允许 | 禁止 |
| 导入/导出/重建记忆 | 禁止 | 禁止 | 允许 | 禁止 |

数据库和审计字段预留：

- `users.role`：用户角色。
- `users.is_terminal_admin`：是否终端管理员，系统内只能有一个 true。
- `strategy_memory_notes.created_by_user_id`：创建人。
- `strategy_memory_notes.created_by_role`：创建角色。
- `strategy_memory_notes.source_type`：manual、ai_reflection、trade_review、ocr_import、pdf_import、video_import。
- `strategy_memory_notes.review_status`：pending、approved、rejected、archived。
- `strategy_memory_notes.reviewed_by_user_id`：审核人，必须为终端管理员。
- `strategy_memory_notes.version`：记忆版本。
- `strategy_memory_audit_logs.action`：view、append、approve、edit、delete、restore、export、import。
- `strategy_memory_audit_logs.ip_address`、`user_agent`、`request_id`：审计追踪字段。

硬性验收：

- 系统启动时必须校验终端管理员唯一性。
- 所有修改、删除、导入、导出 AI 记忆的操作都必须写审计日志。
- 管理账号即使能看，也不能改；普通用户不能看完整记忆。
- AI 自动反思只能进入待审核区，不能直接污染正式策略记忆。
- 任何绕过权限直接写 `strategy_memory.json` 或数据库记忆表的行为，都视为严重安全缺陷。

# 二十一、推荐算法第一版设计

- 没有暴雷。
- 非 ST。

- 低位震荡。
- 到达支撑位。
- 板块热度上升。
- 资金流入。

长线推荐优先找：

- 行业方向较好。
- 公司基本面较稳定。
- 估值不过高。
- 龙头或细分龙头。
- 长期趋势没有破坏。
- 政策支持方向。

## 21.3 概率第一版

概率先用评分映射：

- 90 分以上：75%-85%。
- 80-90 分：65%-75%。
- 70-80 分：55%-65%。
- 60-70 分：50%-55%。

后续用模拟盘结果校准。

# 二十二、模拟盘验收流程

# 二十六、holaOS / 多 Agent 生产级编排

## 26.1 当前真实状态

当前项目里的 Agent 能力还不是生产级 holaOS 编排系统，而是本地 Agent 工作台雏形。

已经具备的内容：

- 前端有 Agent 工作台入口，可以查看数据、新闻、评分、决策、风控、执行等 Agent 的状态。
- 后端已有 `agent_workspace` 本地状态层，会给每个 Agent 写入 `state.json` 和事件日志。
- 系统已经把数据、AI 选股、风控复核、模拟执行、策略记忆等能力按职责拆开。
- 每个 Agent 已有职责边界、状态文件、日志文件和未来交接点。

还没有完成的内容：

- 没有真正启动独立自治 Agent 进程。
- 没有接入 holaOS 的 Harness、任务恢复、全景桌面和生产级调度。
- 没有统一任务队列、事件总线、锁、重试、限流、审计追踪和人工审批流。
- 没有把每个 Agent 的输入、输出、权限、模型路由和失败补偿做成严格生产协议。

所以当前阶段的准确定位是：

> 本地 H5 Agent 工作台 + 状态持久化 + 未来 holaOS 交接文件雏形。

它的价值不是立刻替代生产编排，而是先把系统拆成可观察、可审计、可升级的 Agent 职责单元，避免后续接入 holaOS 时重写核心业务。

## 26.2 这个板块的作用

holaOS / 多 Agent 编排板块的作用，是把“一个大模型聊天式选股工具”升级成“可长期运行、可恢复、可监管、可追责的智能投研与模拟交易操作系统”。

具体作用：

1. 拆职责：让数据、新闻、评分、决策、风控、执行、复盘分别由不同 Agent 负责，避免一个 AI 同时做所有事。
2. 降风险：交易建议必须经过风控 Agent 否决权，执行 Agent 默认只能模拟盘，不允许绕过风控。
3. 可追溯：每一次候选生成、AI 打分、风控否决、模拟下单、复盘学习都留下结构化日志。
4. 可恢复：服务重启后能从上次 Agent 状态继续，不丢任务、不重复下单、不误判持仓。
5. 可扩展：未来可以把轻量任务交给小模型，把决策和风控交给大模型，把回测和数据任务放到异步队列。
6. 可生产化：后续接实盘提醒、半自动确认、机构版时，有明确权限、审批、审计和熔断边界。

一句话：

> 它不是为了“看起来有很多 Agent”，而是为了让 AI 选股和模拟交易从页面功能，升级为长期运行的生产级工作流。

## 26.3 目标生产架构

生产级多 Agent 编排建议采用四层结构：

```text
用户界面 / Agent 工作台
  -> 任务编排层（holaOS Harness / 本地编排器）
  -> Agent 职责层
     - DataAgent
     - NewsAgent
     - TechnicalAgent
     - CapitalAgent
     - SentimentAgent
     - ScoreAgent
     - DecisionAgent
     - RiskAgent
     - ExecutionAgent
     - ReviewAgent
  -> 基础设施层
     - PostgreSQL / SQLite 状态库
     - Redis 队列与缓存
     - JSONL 审计日志
     - 策略记忆库
     - 模型路由与密钥配置
     - 熔断与权限控制
```

第一阶段仍然可以保持单体 FastAPI，但内部必须按这个结构保存任务、状态和日志。等模拟盘稳定后，再把任务编排层替换为 holaOS 或同类生产编排框架。

## 26.4 Agent 职责与权限边界

| Agent | 主要职责 | 允许输出 | 禁止行为 |
| --- | --- | --- | --- |
| DataAgent | 股票池、K线、分时、实时行情、资金流、缓存健康检查 | 标准行情快照、数据质量报告 | 不允许给交易建议 |
| NewsAgent | 新闻、公告、财报风险、政策、情绪事件 | 事件流、风险标签、来源链接 | 不允许把未来新闻写入历史回测 |
| TechnicalAgent | K线形态、支撑压力、趋势、量价结构 | 技术评分和形态证据 | 不允许直接下买卖结论 |
| CapitalAgent | 主力资金、板块资金、龙虎榜、游资行为 | 资金强弱评分 | 不允许单独决定买入 |
| SentimentAgent | 情绪周期、涨停跌停、主线板块、短线温度 | 市场阶段和仓位温度 | 不允许忽略风控直接放大仓位 |
| ScoreAgent | 汇总量化因子、历史有效性、游资经验因子 | 候选池、分项评分、排序依据 | 不允许绕过数据质量检查 |
| DecisionAgent | 生成结构化交易计划 | 买入条件、卖出条件、止损、仓位建议 | 不允许直接执行订单 |
| RiskAgent | 仓位、黑名单、公告暴雷、模型交叉验证、一票否决 | 通过、否决、降级、仅观察 | 不允许被其他 Agent 绕过 |
| ExecutionAgent | 模拟盘订单、成交记录、持仓变化 | 模拟订单、成交流水、失败原因 | 默认不允许真实下单 |
| ReviewAgent | 复盘模拟盘盈亏、总结策略记忆、调整因子建议 | 复盘报告、学习笔记、权重建议 | 不允许未经验证直接改生产权重 |

## 26.5 标准任务流水线

盘后选股流水线：

```text
DataAgent 更新股票池和日K
  -> NewsAgent 更新公告和风险事件
  -> TechnicalAgent / CapitalAgent / SentimentAgent 生成分项证据
  -> ScoreAgent 生成第二阶段候选池
  -> AI 质量打分批处理
  -> DecisionAgent 生成重点复核池和交易计划
  -> RiskAgent 交叉验证并一票否决
  -> ExecutionAgent 写入模拟盘信号
  -> ReviewAgent 次日复盘并沉淀策略记忆
```

盘中盯盘流水线：

```text
DataAgent 刷新实时行情
  -> TechnicalAgent 判断触发条件
  -> RiskAgent 检查熔断、仓位、公告风险
  -> DecisionAgent 更新买卖提醒
  -> ExecutionAgent 仅执行模拟盘或发出人工确认提醒
```

## 26.6 状态、日志和交接协议

每个 Agent 必须有独立状态文件和日志：

```text
backend/data/agents/{agent_id}/state.json
backend/data/agents/{agent_id}/events.jsonl
backend/data/agents/{agent_id}/artifacts/
```

`state.json` 至少包含：

- `agent_id`
- `status`
- `current_task_id`
- `last_success_at`
- `last_error`
- `input_version`
- `output_version`
- `risk_lock`
- `handoff_to`
- `human_approval_required`

`events.jsonl` 每行记录：

- 时间
- Agent
- 动作
- 输入摘要
- 输出摘要
- 失败原因
- 是否影响交易建议
- 是否触发风控

这样做的目的，是让每个交易建议都能回答三个问题：

1. 它为什么出现？
2. 谁批准或否决了它？
3. 出错后从哪里恢复？

## 26.7 模型路由与成本控制

不同 Agent 不需要使用同一个模型：

- 轻量任务：数据整理、格式校验、日志摘要，使用小模型或纯规则。
- 中量任务：技术面、资金面、新闻分类，使用中等模型或本地算法。
- 重量任务：最终交易计划、风控复核、异常解释，使用强模型。
- 高风险任务：买入建议、仓位建议、公告暴雷判断，必须支持主模型和独立风控模型交叉验证。

模型路由必须记录到日志中，包括模型名、任务类型、超时、是否失败、是否使用兜底结果。

## 26.8 风控与人工审批

生产级编排必须遵守以下硬规则：

1. RiskAgent 拥有最终否决权。
2. ExecutionAgent 默认只允许模拟盘。
3. 真实交易接口默认关闭。
4. 任何真实下单能力必须经过人工确认、权限校验、风控通过和熔断检查。
5. 模型超时、返回非 JSON、风险数据缺失时，只能降级为观察或失败，不能伪装成高置信推荐。
6. 策略记忆只能作为建议和上下文，不能未经回测直接改实盘权重。

## 26.9 分阶段落地路线

### Phase A：本地 Agent 工作台补完

目标：把当前雏形做扎实。

- 修正 Agent 元数据中文显示。
- 为每个 Agent 补齐状态文件、事件日志和 artifact 输出。
- 前端工作台显示每个 Agent 的最近任务、最近错误、数据新鲜度和风控状态。
- AI 选股、智能筛选、公告回填、模拟盘交易都写入对应 Agent 日志。

验收标准：

- 打开工作台能看到每个 Agent 当前在做什么。
- 任意一条 AI 推荐都能追溯到候选池、AI 打分、风控复核和模拟信号。

### Phase B：本地生产编排器

目标：在不接 holaOS 的情况下先实现生产编排基本能力。

- 新增统一 `task_id`。
- 新增任务队列、任务状态、重试、超时、失败补偿。
- 后台任务不再散落在接口函数里，而是统一进入编排器。
- 每个任务明确输入、输出、依赖、失败处理和是否需要人工确认。

验收标准：

- 服务重启后任务状态可恢复。
- 失败任务能看到失败原因和可重试入口。
- 不会重复生成买入信号或重复模拟下单。

### Phase C：holaOS 接入适配

目标：把本地 Agent 状态和任务协议映射到 holaOS Harness。

- 将本地 `state.json`、`events.jsonl`、artifact 目录作为 holaOS Agent handoff 协议。
- 将 DataAgent、RiskAgent、ExecutionAgent 等拆成可独立运行的任务单元。
- 接入 holaOS 的桌面视图、状态恢复和多 Agent 调度能力。
- 保留 FastAPI 作为业务 API，不把交易业务强绑定在编排框架内部。

验收标准：

- holaOS 能看到每个 Agent 的状态、任务、日志和交接结果。
- 任一 Agent 重启后能从上次状态恢复。
- 风控 Agent 的否决结果能阻断后续执行。

### Phase D：生产级交易辅助

目标：面向长期运行和多人使用。

- 加入权限、审计、人工审批和操作确认。
- 加入模型费用统计和任务限流。
- 加入异常监控、熔断通知和日终报告。
- 支持多人、多模拟盘、多策略组合。
- 真实交易仍默认关闭，只作为独立受控能力预留。

验收标准：

- 连续运行 20 个交易日，任务、日志、模拟交易、复盘均可追溯。
- 任意推荐和任意模拟订单都能完整还原决策链。
- 风控异常时系统自动停止推荐买入或降级为观察。

## 26.10 当前优先级

短期不建议马上重度接入 holaOS。当前更合理的顺序是：

1. 先把本地 Agent 工作台补完整。
2. 再把 AI 选股、公告缓存、智能筛选、模拟交易全部接入统一任务日志。
3. 然后做本地编排器，解决恢复、重试、审计和防重复执行。
4. 最后接 holaOS，让它承接多 Agent 桌面、长期运行和生产级调度。

这样做风险最低，也最符合当前项目已经形成的业务闭环。

# 二十七、总结

当前阶段最正确的做法是：

> 先做一个小而完整、能实时模拟交易、能让你亲自用模拟盘验收的小程序。

第一版不要做复杂账户、收费、实盘、托管、机构后台。核心只做：

1. 股票数据。
2. 推荐筛选。
3. 长短线推荐。
4. 小程序展示。
5. 实时模拟买卖。
6. 持仓盈亏。
7. 交易记录。
8. 模拟盘复盘。

只要模拟盘实操符合你的预期，后续再逐步升级到更完整的平台。这样开发成本最低、验证速度最快、风险最小，也最符合当前只有两三个人使用的实际情况。
